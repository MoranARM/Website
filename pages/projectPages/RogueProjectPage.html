Rogue-like Dungeon Crawler Project

Project Summary:

I built a text based Rougue-like Dungeon Crawler in C++ as an individual class project. My task involved inputting a file containing room and item data, then parsing the data into objects using polymorphism and allowing the user to navigate the dungeon. The final deliverable allows the user to enter a text file then move between rooms and use items to escape the dungeon. The project was completed across two weeks, with file loading and room navigation done in the first week, and item addition and use completed in week two. What I found the most challenging about this project was deciding the most efficient way to handle input from the console utilizing polymorphism and object-oriented programming. Knowing what parts of the super class should be virtual and what parts should have their own implementation for the subclasses to use was somewhat difficult in trying to find a nice balance where code is reused, and not copied. Thus, I ultimately decided it was logical for the maze to call on implemented virtual methods from room, and bag (both of which are subclasses of collection) since that would allow transfer between the two locations for picking up and dropping items.


Project Analysis:

Things that I did well in the project included optimization code reuse using polymorphism. Without proper optimization there would have been a significant amount of repeated code for each item added, along with the inefficient storage of those items without implementing collection on each potential storage. These optimizations also allowed for easier manipulation of items since they were all reference-type item they could be held and manipulated within one data structure.

New skills that I learned from working on the project include how polymorphism is done in C++ with the virtual keyword and including = 0 at the end of the virtual method in the header file means that it is entirely abstract. I also learned how memory management of classes is done with destructors and deleting pointers to other variables within the destructor. The use of subclasses that extend from an abstract superclass proved useful in how items can be manipulated the same despite there being no strictly item class objects, since it makes a lot more sense to have the specific kind of item as its own class. Old skills that I revisited included the use of polymorphism with abstract classes, and taking user input as I had only previously implemented those in Java and javascript. Another revisited skill was reading from a file using fstream as an ifstream object and reading in with the getline function.

What went well included the navigation between non-euclidian rooms, especially though the use of teleportation items consumed on use. One major opportunity for improvement that I identified for my next project was altering the need for a user to upload their own dungeon file, where a default, or even multiple levels could be provided by the program. This would eliminate the need for uploading files, although it would be nice to keep as a feature, the total dungeon size would increase greatly from the multiple floors that the user needs to move between. Another similar project that I worked on could benefit from the use of loading in files as a map was a pac-man AI that I wrote in java, and javascript. The use of custom maps here would challenge the AI to be more versatile and not as dependent on the structure of the default map.
